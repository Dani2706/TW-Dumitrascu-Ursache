Document de specificare a cerințelor principale - REM

Autori

  • Dumitrașcu Ștefan Daniel
  • Ursache Ștefan

1. Introducere

1.1 Scop

Acest document stabilește cerințele pentru o platformă imobiliară care le oferă utilizatorilor posibilitatea de a explora, căuta și publica anunțuri de proprietăți. El include o descriere amplă a funcționalităților sistemului, a limitărilor sale și a modului în care acesta interacționează cu utilizatorii și alte sisteme.

1.2 Sfera produsului

REM este o aplicație web care permite proprietarilor să își listeze proprietățile pentru vânzare sau închiriere, în timp ce potențialii cumpărători sau chiriași pot căuta și naviga prin aceste anunțuri. Sistemul oferă, de asemenea, funcționalități administrative pentru gestionarea utilizatorilor și a proprietăților.

1.3 Definiții, acronime și prescurtări

SRS
Software Requirements Specification
UI
User Interface
API
Application Programming Interface
DTO
Data Transfer Object
JWT
JSON Web Token

2. Descriere generală

2.1 Perspectiva produsului

REM este o aplicație web independentă, construită pe o arhitectură client-server. Aceasta include un backend Java bazat pe servlete și un frontend JavaScript cu arhitectură bazată pe componente.

2.2 Funcțiile produsului

  • Înregistrarea și autentificarea utilizatorilor
  • Gestionarea proprietăților listate (creare, vizualizare, actualizare, ștergere)
  • Căutarea și filtrarea proprietăților
  • Gestionarea anunțurilor favorite
  • Administrarea profilului utilizatorului
  • Funcții administrative pentru gestionarea utilizatorilor și a proprietăților
  • Statistici privind proprietățile listate

2.3 Caracteristicile utilizatorilor

Sistemul suportă două tipuri principale de utilizatori:

  • Utilizatori: Orice persoană înregistrată în sistem poate folosi aplicația pentru a cumpăra, închiria sau pentru a oferi proprietăți spre vânzare și/sau închiriere. Astfel, un utilizator poate avea unul sau mai multe roluri active simultan: cumpărător, chiriaș sau proprietar.
  • Administratori: Au acces complet la funcțiile de gestionare a utilizatorilor și a proprietăților.

2.4 Constrângeri

  • Sistemul trebuie să fie accesibil prin intermediul browserelor web.
  • Backend-ul este dezvoltat în Java, iar frontend-ul în JavaScript.
  • Autentificarea utilizatorilor se realizează folosind JWT.
  • Sistemul trebuie să gestioneze eficient imaginile proprietăților.
  • Designul sistemului trebuie să fie responsive, adaptându-se la diferite dimensiuni de ecran.

4. Arhitectura Sistemului

4.1 Diagrama componentelor

Aplicația urmează o arhitectură în trei niveluri:

  • Stratul de prezentare: Interfață frontend bazată pe JavaScript, cu arhitectură componentă
  • Stratul de aplicație: Servlete Java care gestionează logica și punctele finale API
  • Stratul de date: Gestionarea bazei de date și obiectele de acces la date

4.2 Diagrama de implementare

Aplicația este implementată astfel::

  • Partea client: Browsere web care accesează HTML, CSS și JavaScript
  • Partea server: Aplicație Java rulând într-un container de servlete
  • Serverul de baze de date: Bază de date relațională pentru persistența datelor

5. Detalii despre Implementare

5.1 Implementarea Frontend-ului

Interfața frontend este realizată folosind o arhitectură bazată pe componente, cu JavaScript simplu. Componentele principale includ:

  • HomeComponent: Pagina principală a aplicației
  • LoginComponent și RegisterComponent: Interfețe pentru autentificarea utilizatorilor
  • PropertiesListComponent: Afișează lista de proprietăți
  • PropertyComponent: Vizualizare detaliată a unei singure proprietăți
  • AddPropertyComponent și EditListingComponent: Formulare pentru gestionarea proprietăților
  • FavoriteListingsComponent: Gestionarea proprietăților favorite ale utilizatorului
  • AdminPropertyDashboardComponent și AdminUserDashboardComponent: Interfețe administrative

5.1.1 Detalii de Interfață Utilizator

Interfața utilizator implementează următoarele caracteristici pentru a asigura o experiență optimă:

Navigare și Interacțiune
  • Tranziții graduale între paginile principale și la selectarea categoriilor de anunțuri
  • Iconițe vizuale dedicate fiecărei categorii de proprietăți pentru orientare rapidă
  • Componente ascunse inițial (precum butoanele de schimbare a categoriei) și afișate doar când este necesar
Design și Feedback
  • Paletă cromatică dominată de albastru închis (#003366) pentru transmiterea valorilor de încredere și profesionalism
  • Mesaje de încărcare clare în timpul procesării datelor din backend
  • Mesaje personalizate pentru situațiile fără rezultate (căutări nereușite, conturi fără anunțuri)
  • Validare în timp real pentru câmpurile formularelor, cu mesaje de eroare contextuale
Funcționalități avansate
  • Hartă interactivă cu marcarea proprietăților sub formă de pin-uri
  • Actualizare automată a hărții în timp real la filtrarea sau căutarea proprietăților
  • Funcționalitate de delimitare a zonei de interes direct pe hartă prin desenarea unui poligon
  • Câmpuri de selecție dinamice pentru oraș și stat, afișând doar valorile relevante pentru proprietățile curente
  • Sistem de sortare flexibil și căutare în timp real, fără necesitatea apăsării unui buton suplimentar
Experiență completă
  • Anunțuri cu imagine principală reprezentativă, titlu, preț și caracteristici secundare relevante
  • Funcționalitate de adăugare la favorite cu verificare automată a autentificării
  • Sistem complet de căutare și sortare în pagina de anunțuri proprii

5.2 Implementarea Backend-ului

Backend-ul este dezvoltat folosind Java Servlets. Componentele principale includ:

  • Controlere: Servlete care gestionează diferite cereri HTTP
  • DTO-uri: Obiecte pentru transferul datelor între straturi
  • Entități: Modele de domeniu care reprezintă tabelele din baza de date
  • Servicii: Implementarea logicii de business
  • Repository-uri: Strat de acces la date pentru operațiuni pe baza de date
  • Filtre: Filtre pentru procesarea cererilor, responsabile de autentificare și autorizare

5.3 Proiectarea Bazei de Date

Schema bazei de date include următoarele entități principale:

  • Users: Stochează informațiile și credențialele utilizatorilor
  • Admin: Reține toți utilizatorii cu un câmp care indică statusul de administrator
  • Properties: Stochează detaliile proprietăților listate
  • PropertyMainImage: Stochează imaginile principale ale proprietăților
  • PropertyImages: Stochează imaginile suplimentare pentru proprietăți
  • Favorites: Leagă utilizatorii de proprietățile lor favorite
  • Views: Monitorizează numărul de vizualizări ale proprietăților
  • TopProperties: Reține anunțurile considerate „de top”, pe baza numărului de vizualizări
  • TopUsersHistory: Păstrează istoricul utilizatorilor care au avut anunțuri în lista TopProperties

5.4 Securitate

Securitatea aplicației este implementată prin mai multe mecanisme:

  • Autentificare bazată pe JWT:
    • Utilizatorii primesc un token JWT după autentificarea cu succes (LoginController generează acest token)
    • Tokenul conține informații despre utilizator (username, userId și adminStatus), gestionate prin DataForJwtCreationDTO
    • Clasa JwtUtil oferă funcționalități pentru generarea, validarea și verificarea rolurilor din token
    • Tokenul este transmis în header-ele HTTP pentru autentificarea ulterioară a cererilor
  • Controlul accesului bazat pe roluri:
    • Există două roluri principale: utilizatori standard și administratori
    • Statutul de administrator este verificat prin metoda isAdmin() din clasa JwtUtil
    • Anumite rute și funcționalități sunt disponibile doar pentru administratori
    • Clientul primește informația despre rolul utilizatorului (isAdmin) în timpul autentificării
  • Protecția rutelor:
    • Rutele API sunt protejate prin verificarea prezenței și validității tokenului JWT
    • Rutele administrative sunt accesibile doar utilizatorilor cu statut de administrator
    • Rutele pentru editarea și ștergerea anunțurilor sunt restricționate la proprietarul anunțului sau administratori
  • Redirecționare SPA:
    • SpaRedirectFilter asigură că toate cererile care nu sunt pentru resurse statice sau API sunt redirecționate către pagina principală
    • Acest filtru permite rutarea pe partea de client pentru aplicația SPA, menținând securitatea navigației
  • Securitatea parolelor:
    • Parolele utilizatorilor sunt stocate în format hash, nu în text clar
    • Validarea parolelor se face prin compararea hash-urilor, nu a parolelor în text
  • Gestionarea erorilor de autentificare:
    • Erorile de autentificare sunt gestionate explicit (vezi InvalidUsernameException și InvalidPasswordException)
    • Mesajele de eroare sunt generale pentru a evita divulgarea informațiilor sensibile
    • Evenimentele de autentificare eșuată sunt înregistrate în jurnal pentru detectarea potențialelor atacuri
  • Securitatea API:
    • Validarea și sanitizarea intrărilor pentru a preveni atacurile de injecție
    • Folosirea tipelor de conținut adecvate (application/json) pentru comunicarea client-server
    • Codificarea caracterelor adecvată (UTF-8) pentru a preveni probleme de securitate legate de codificare

Aceste măsuri de securitate lucrează împreună pentru a proteja datele utilizatorilor și a asigura că numai utilizatorii autorizați pot accesa și modifica informațiile din sistem.

6. Concluzii

Acest document oferă o specificație detaliată a cerințelor pentru Platforma de Anunțuri Imobiliare. El servește drept ghid pentru dezvoltare și testare, asigurând o înțelegere clară a funcționalităților și constrângerilor sistemului pentru toți factorii implicați.
